# Project Context Runtime Implementation

This document records the current implementation of the OpenCode Project Context runtime. It is intended as the baseline for optimizing automatic `update` latency without changing behavior.

Source of truth:

- Maintainer agent configuration: `src/integrations/opencode/config-hook.ts`
- Maintainer system prompt: `src/integrations/opencode/maintainer-prompt.ts`
- Maintainer session / subtask prompt rendering: `src/integrations/opencode/subsession-runner.ts`
- Update payload construction and path format: `src/integrations/opencode/auto-update-hook.ts`, `src/integrations/opencode/auto-update-payload.ts`
- Scaffold constants, template, validation, and parsing: `src/core/constants.ts`, `templates/prjctxt-template/*`, `src/workspace/bootstrap.ts`, `src/indexer/parser.ts`, `src/capsule/context-capsule.ts`

## 1. Maintainer agent configuration

The hidden maintainer agent ID is:

```text
project-context-maintainer
```

The agent is injected into OpenCode config by `createProjectContextConfigHook()`.

### 1.1 Agent shape

```ts
{
  mode: "subagent",
  hidden: true,
  description: "Internal project context maintainer. Invoked only by the Project Context runtime.",
  prompt: buildMaintainerPrompt(),
  permission: { ... },
  tools: { ... }
}
```

The maintainer is not a user-facing agent. It is intended to be invoked only by the Project Context runtime.

### 1.2 Enabled and disabled tools

The maintainer tool surface is explicitly constrained.

Enabled:

```ts
read: true
glob: true
edit: true
write: true
patch: true
apply_patch: true
bash: true
```

Disabled:

```ts
webfetch: false
websearch: false
task: false
delegate_task: false
delegate_tesk: false
project_context_search: false
session: false
session.create: false
session.prompt: false
session.promptAsync: false
```

The `delegate_tesk` entry intentionally exists alongside `delegate_task` as a defensive spelling-compatible deny entry.

### 1.3 Read permissions

Maintainer `read` is scoped to the private Project Context workspace plus the update job cache.

Allowed project-context paths:

```ts
.crewbee/.prjctxt/**
**/.crewbee/.prjctxt/**
.crewbee\.prjctxt\**
**\.crewbee\.prjctxt\**
```

Additional allowed runtime payload path:

```ts
.crewbee/.prjctxt/cache/update-jobs/**
```

Default read policy:

```ts
"*": "deny"
```

### 1.4 Write permissions

The write-like tools `edit`, `write`, `patch`, and `apply_patch` share the same scoped permission object.

Allowed paths:

```ts
.crewbee/.prjctxt/**
**/.crewbee/.prjctxt/**
.crewbee\.prjctxt\**
**\.crewbee\.prjctxt\**
```

Default write policy:

```ts
"*": "deny"
```

This means the maintainer may update only the private Project Context scaffold and must not modify product code.

### 1.5 Bash permissions

Allowed bash commands:

```ts
git status
git status *
git diff
git diff *
git log
git log *
npm run doctor
npm run doctor *
```

All other bash commands are denied:

```ts
"*": "deny"
```

### 1.6 Explicit denied permissions

The maintainer explicitly denies:

```ts
webfetch
websearch
project_context_search
session
session.create
session.prompt
session.promptAsync
```

### 1.7 Restrictions applied to other agents

For non-maintainer agents:

- Primary/all-mode agents are denied direct task invocation of `project-context-maintainer`.
- Subagents are denied `project_context_search` and have that tool disabled.
- Defensive runtime session tools are denied for Project Context runtime access.

### 1.8 Watcher ignore entries

The config hook appends watcher ignore entries for runtime/private transient files:

```text
.crewbee/.prjctxt/cache/**
.crewbee/.prjctxt/tmp/**
.crewbee/.prjctxt/*.lock
```

The same entries are also added through `PRIVATE_RUNTIME_CONTEXT_DIR`, which currently equals `.crewbee/.prjctxt`.

## 2. Maintainer system prompt

The maintainer system prompt is generated by `buildMaintainerPrompt()`.

Current prompt text:

```text
You are Project Context Maintainer.

You maintain the .crewbee/.prjctxt/ project context workspace for OpenCode + CrewBee projects.
You are not user-facing. You are invoked only by the Project Context runtime for initialization, search, and automatic update jobs.

Responsibilities:
- Initialize new project context workspaces by reading project documentation, architecture/design notes, package metadata, tests, and main source implementation.
- Prepare compact project context for the main agent.
- Search prior project context when requested.
- Maintain framework design, implementation snapshot, plan, decisions, risks, and handoff.
- Keep .crewbee/.prjctxt/ compact, accurate, and actionable.

Rules:
- Never call project_context_search or any Project Context runtime tool. Runtime orchestration owns initialize, search, and update jobs.
- Never create OpenCode sessions or prompt sessions. The plugin runtime owns maintainer session orchestration.
- For automatic update jobs, first use the read tool (not bash/shell) to read the runtime payload JSON from .crewbee/.prjctxt/cache/update-jobs/<jobID>.json when the prompt provides a Job ID. The parent session prompt intentionally contains only the Job ID; the payload contains the parent summary, assistant final text, changed files, diff summary, and verification summary.
- Do not expose .crewbee/.prjctxt/ file structure to the main agent.
- Do not ask the main agent to choose context files.
- Do not modify product code.
- Only edit .crewbee/.prjctxt/** when an initialization or automatic update job requires context maintenance.
- Never fabricate verification.
- Keep outputs compact.
- Treat MEMORY_INDEX as high-signal only.
- Write HANDOFF.md for the next session, not as a long report.
- If uncertain, preserve current context and report a warning.
- At the end of every initialize job, explicitly output a compact change summary with: files changed, decisions/state updated, verification recorded, and blockers/warnings. If no files changed, say so and explain why.

Automatic update job payloads:
- For project_context_update jobs, the task prompt includes only a Job ID. Read the private payload JSON from .crewbee/.prjctxt/cache/update-jobs/<jobID>.json.
- Read that JSON file before updating context; it contains the parent session summary, assistant final text, changed files, git status/diff summaries, verification outputs, decisions, blockers, and next actions.
- Use the job payload as the source of truth for the main-session turn, then inspect current repo state if needed.
- Run Project Context doctor/consistency checks when available.
- Do not expose private scaffold paths, changed scaffold files, payload content, decisions, diff summaries, verification output, or blockers in your final response.
- Your final project_context_update response must be exactly one compact sentinel line: PROJECT_CONTEXT_UPDATE_DONE job=<jobID> status=<ok|failed>
- The Project Context runtime deletes the job payload file after the update Task completes.
```

## 3. Maintainer update trigger prompt

Automatic update runs are submitted by `MaintainerSubsessionRunner`.

### 3.1 Update fast path: parent session subtask card

When `job.kind === "update"` and the OpenCode client exposes `session.prompt`, the runtime submits a subtask part to the parent session instead of creating a separate session manually.

Submitted part shape:

```ts
{
  type: "subtask",
  prompt: renderJob(job),
  description: job.title,
  agent: "project-context-maintainer"
}
```

The `session.prompt` body intentionally does not include `noReply`.

The runtime observes submission for up to `UPDATE_SUBTASK_SUBMIT_OBSERVE_MS`, currently `1000` ms. If submission remains pending, the runtime logs the pending state and returns successfully so the parent session is not blocked by the maintainer run.

### 3.2 Current update task prompt text

For update jobs, `renderJob(job)` returns:

```text
Project Context Maintainer job: update
Job ID: <jobID>

Load the update payload from the private Project Context job store.
Use the payload as the source of truth for this update.
Update only the private Project Context scaffold.
Do not modify product code.
Do not expose private scaffold paths in user-facing output.

Completion protocol:
- After the scaffold update and checks are complete, do not summarize details.
- Do not list changed scaffold files.
- Do not include decisions, diff summaries, verification output, or private paths.
- Your final response must be exactly one compact sentinel line:
PROJECT_CONTEXT_UPDATE_DONE job=<jobID> status=<ok|failed>
```

Only the Job ID is exposed in the parent-session subtask prompt. The payload path and payload content are intentionally private and are not included in the parent prompt.

### 3.3 Fallback path: explicit maintainer session

For non-update jobs, or when the update fast path is unavailable, the runtime creates a maintainer session and sends `renderJob(job)` via `session.promptAsync`.

In this fallback prompt path, the prompt body includes:

```ts
agent: "project-context-maintainer"
tools: {
  project_context_search: false,
  task: false,
  delegate_task: false,
  delegate_tesk: false
}
parts: [{ type: "text", text: renderJob(job) }]
```

The runner refuses to use blocking `session.prompt` as a fallback from a tool execution callback.

### 3.4 Maintainer timeouts

Default timeouts by job kind:

```ts
initialize: 180_000 ms
search: 45_000 ms
update: 90_000 ms
```

The timeout may be overridden through `CREWBEE_PROJECT_CONTEXT_MAINTAINER_TIMEOUT_MS` when set to a positive finite number.

Polling interval:

```ts
500 ms
```

Individual OpenCode API calls are capped at:

```ts
15_000 ms
```

## 4. Update payload design

The automatic update runtime writes a private JSON payload before submitting the maintainer update task.

### 4.1 Payload ID and path

Job ID format:

```text
pcu_<Date.now().toString(36)>_<random-base36-8-chars>
```

Payload path:

```text
<projectRoot>/.crewbee/.prjctxt/cache/update-jobs/<jobID>.json
```

### 4.2 Payload schema

The current payload shape is:

```ts
interface UpdateJobPayload {
  schemaVersion: 1;
  kind: "project_context_update";
  jobID: string;
  createdAt: string;
  trigger: {
    reasons: string[];
    toolEvents: RecordedToolEvent[];
  };
  parentSession: {
    id: string;
    latestUserRequest: string;
    assistantFinalText: string;
    decisionsDetected: string[];
    nextActionsDetected: string[];
    blockersDetected: string[];
  };
  engineeringChanges: {
    changedFiles: string[];
    gitStatusSummary: string;
    gitDiffSummary: string;
    verification: RecordedToolEvent[];
  };
  instruction: string[];
}

interface RecordedToolEvent {
  tool: string;
  reason: string;
  argsSummary: string;
  resultSummary?: string;
  timestamp: string;
}
```

### 4.3 Git data collected for payload

The runtime collects:

```text
git status --short
git diff --stat
git diff --cached --stat
git diff --name-only
git diff --cached --name-only
```

Changed files are derived from unstaged diff names, staged diff names, and `git status --short` paths.

### 4.4 Payload cleanup

The payload is tracked in runtime maps and cleaned up when:

- the maintainer reads the payload with the `read` tool;
- the maintainer session becomes idle;
- the maintainer run completes or fails;
- the parent-side update run fails after payload registration;
- the retention timer expires.

Retention timer:

```text
30 minutes
```

## 5. Scaffold design

The private scaffold root is:

```text
.crewbee/.prjctxt
```

`PRIVATE_RUNTIME_CONTEXT_DIR` currently equals `DEFAULT_CONTEXT_DIR`, so runtime payloads also live under `.crewbee/.prjctxt`.

Initialization creates:

```text
.crewbee/.prjctxt/
.crewbee/.prjctxt/observations/
.crewbee/.prjctxt/cache/
```

It then writes template files from `templates/prjctxt-template/`, replacing:

```text
new-project -> projectId
New Project -> projectName
```

Unless forced, existing scaffold files are skipped.

### 5.1 Required files

The required scaffold files are:

```text
QUICKSTART.md
PROJECT.md
ARCHITECTURE.md
IMPLEMENTATION.md
PLAN.yaml
STATE.yaml
HANDOFF.md
MEMORY_INDEX.md
DECISIONS.md
REFERENCES.md
config.yaml
```

### 5.2 Searchable files

The searchable/prepare-relevant context files are:

```text
PROJECT.md
ARCHITECTURE.md
IMPLEMENTATION.md
PLAN.yaml
STATE.yaml
HANDOFF.md
MEMORY_INDEX.md
DECISIONS.md
REFERENCES.md
```

`QUICKSTART.md` and `config.yaml` are required, but not part of `SEARCHABLE_CONTEXT_FILES`.

### 5.3 Template file structures

#### QUICKSTART.md

Template structure:

```md
# New Project Context Quickstart

Project Context initialization, prepare, and update are automatic. Use `project_context_search` only as a rare fallback when auto init/prepare/update still leave a concrete historical project-context gap that blocks the task.

On first startup, the plugin creates this scaffold if required files are missing and delegates the hidden maintainer to initialize it from project docs, architecture/design notes, tests, package metadata, and main source implementation.
```

Purpose: onboarding note for the private scaffold.

#### PROJECT.md

Template structure:

```md
# Project

## Project ID

new-project

## Project Name

New Project

## Objective

Describe the project objective.

## Constraints

- Follow the minimalism principle.
- Do not store secrets in .crewbee/.prjctxt/.
```

Used by prepare to extract project name when extended project context is included.

#### ARCHITECTURE.md

Template structure:

```md
# Architecture

## System Map

TBD
```

Extended prepare may read architecture highlights from sections:

```text
System Map
Module Responsibilities
Key Invariants
Failure Handling
```

Only `System Map` exists in the starter template.

#### IMPLEMENTATION.md

Template structure:

```md
# Implementation Snapshot

## What Works

TBD

## Verification Commands

TBD
```

Prepare reads implementation highlights from sections:

```text
What Works
Important Paths
Known Gaps
Runtime Flow
Last Verified
```

Validation warns if `## Verification Commands` is absent.

#### PLAN.yaml

Template structure:

```yaml
project_id: new-project
cycles:
  - id: C1
    title: Initial cycle
    status: active
    steps:
      - id: S1
        title: Define project context
        status: active
```

Validation requires:

- `project_id` scalar;
- at least one cycle ID;
- at least one step ID;
- no duplicate step IDs.

The parser reads cycle IDs from lines matching two-space `- id:` and step IDs from lines matching six-space `- id:`.

#### STATE.yaml

Template structure:

```yaml
project_id: new-project
run_status: running
active_cycle: C1
active_step_id: S1
last_checkpoint: CP-0001
blockers: []
next_actions:
  - action: Fill in project context files
    owner: active-agent
    source: PLAN.yaml
```

Validation requires scalar values for:

- `project_id`
- `run_status`
- `active_cycle`
- `active_step_id`

Validation also requires:

- `STATE.yaml project_id` matches `PLAN.yaml project_id`;
- `active_cycle` exists in `PLAN.yaml`;
- `active_step_id` exists in `PLAN.yaml`.

Validation warns when `next_actions` has no `- action:` entries.

#### HANDOFF.md

Template structure:

```md
# Session Handoff

## Current Snapshot

- Active step: C1/S1.

## Exact Next Actions

1. Fill in project context files.
```

Validation requires:

- `## Current Snapshot` section;
- non-empty `## Exact Next Actions` list items.

Prepare also reads optional `## Open Blockers` list items.

#### MEMORY_INDEX.md

Template structure:

```md
# Memory Index
```

The parser expects optional memory blocks split by blocks beginning with:

```text
- ID:
```

Within each block it recognizes:

```text
ID:
Type:
Summary:
```

Prepare ranks and includes up to the configured memory limit.

#### DECISIONS.md

Template structure:

```md
# Decisions
```

Extended prepare reads recent non-heading bullet/text lines as decision highlights.

#### REFERENCES.md

Template structure:

```md
# References
```

References are searchable but are not currently surfaced directly in the compact prepare brief.

#### config.yaml

Template structure:

```yaml
version: 1
primer:
  budget_tokens: 1000
  memory_limit: 5
write_policy:
  mode: tool-confirmed
```

This file is required as scaffold metadata. The current prepare implementation uses hardcoded prepare budgets in `ProjectContextService` rather than reading these values directly.

### 5.4 Prepare brief consumption of scaffold

Compact prepare reads:

- `STATE.yaml`
- `PLAN.yaml`
- `HANDOFF.md`
- `IMPLEMENTATION.md`
- `MEMORY_INDEX.md`

Normal/extended prepare additionally includes:

- `PROJECT.md`
- `ARCHITECTURE.md`
- `DECISIONS.md`

Prepare brief sections include:

- current active step, status, checkpoint, blockers;
- plan window around the active step;
- current snapshot from `HANDOFF.md`;
- implementation highlights;
- optional architecture highlights;
- optional recent decisions;
- exact next actions;
- high-signal memory.

Private scaffold paths are sanitized in generated prepare text.

## 6. Current optimization-relevant observations

The current update path has several time-cost components that are useful for future optimization analysis:

1. Update triggering waits until the parent session is idle and requires a current-turn `files_changed` tool event.
2. Before writing the payload, the runtime reads recent parent session messages.
3. Payload construction runs multiple git commands.
4. The runtime writes a payload JSON file under the private scaffold cache.
5. The runtime submits a maintainer subtask to the parent session and observes submission for up to 1 second.
6. Actual scaffold update work is delegated to `project-context-maintainer` through the subtask prompt.

This document intentionally records current behavior only. It does not propose or implement latency optimizations.
